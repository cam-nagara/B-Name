# CLIP 形式書き出し 見送り計画書

作成日: 2026-04-26  
セッション種別: 調査セッション  
方針: **現時点では実装しない。判断材料と再開手順のみ残す。**

## 1. 今回の結論

B-Name から CLIP STUDIO PAINT の `.clip` を直接書き出す案は、**いったん見送る**。

見送り時点での到達目標は次のとおり:

- B-Name の **コマ** を CSP の **コマフォルダ** として出力したい
- B-Name の **Grease Pencil** を CSP 内では **ラスターレイヤー** として扱えれば十分
- **フキダシ / テキスト / 効果線** も同様に、必要ならラスターレイヤーでよい

今回は実装しないが、将来再開するなら **「コマフォルダ + ラスターレイヤー群」までを現実的なゴール** とする。  
**GP を CSP ネイティブのベクターレイヤーへ変換する案は対象外** とする。

## 2. 見送り理由

### 2.1 `.clip` は単純な SQLite ではない

サンプル `page0017.clip` の先頭は `SQLite format 3` ではなく `CSFCHUNK` だった。  
`.clip` 全体は独自チャンクコンテナで、その内部に `CHNKSQLi` として SQLite 本体があり、さらに `CHNKExta` 群に外部 BLOB が分離されている。

つまり、単に SQLite を生成するだけでは足りない。  
少なくとも以下を整合させる必要がある:

- CSFCHUNK ファイルヘッダ
- `CHNKHead`
- `CHNKExta` 群
- `CHNKSQLi` 内の SQLite
- SQLite 内の `ExternalChunk` 参照

### 2.2 コマフォルダは「ベクター枠」ではなく実質ラスタ枠

実サンプルでは、CSP のコマフォルダは次の条件で識別できた。

- `LayerFolder = 1`
- `ComicFrameLineMipmap != 0`

この `ComicFrameLineMipmap` は `Mipmap` → `Offscreen` → `CHNKExta` に繋がり、  
コマ枠線画像は **ラスタデータ** として保持されていた。

つまり B-Name 側のコマ形状をそのまま「CSP独自ベクター枠」として入れるのではなく、  
**コマ枠をラスタ化してコマフォルダに持たせる** 前提で考えるべき。

### 2.3 GP の CSP ベクター化は危険

実サンプルでは、CSP のベクターレイヤーは `VectorObjectList.VectorData` から外部 BLOB を参照していた。  
この BLOB 形式は公開仕様が不完全で、安定した書き込み仕様として扱えない。

そのため、以下の方針に固定する:

- **GP → CSPベクターレイヤー** は狙わない
- **GP → ラスターレイヤー** で十分とする

### 2.4 B-Name 側の現状でも、まずは内部ラスタ書き出し基盤が必要

現状の B-Name の書き出しは、主にコマの `_preview.png / _thumb.png` を貼り込む段階で止まっている。  
Grease Pencil、フキダシ、テキスト、効果線の書き出しはまだ統合されていない。

したがって、`.clip` 直書きに着手する前に、まず B-Name 内部で

- GP
- フキダシ
- テキスト
- 効果線

を **ページ単位 / コマ単位でラスタ化できること** が前提になる。

## 3. 再開時のゴール定義

再開時の実装ゴールは次の 2 段階に分ける。

### 3.1 現実的ゴール

- コマを CSP の **コマフォルダ**
- GP を **ラスターレイヤー**
- フキダシを **ラスターレイヤー**
- テキストを **ラスターレイヤー**
- 効果線を **ラスターレイヤー**

として `.clip` に出力する。

### 3.2 非ゴール

以下は今回の計画から外す。

- GP を CSP ネイティブのベクターレイヤー化
- フキダシを CSP の専用オブジェクト化
- 効果線を CSP の専用編集可能オブジェクト化
- 3D レイヤーや定規データの再構成

## 4. B-Name → CSP の想定マッピング

### 4.1 コマ

B-Name の各コマは CSP では **コマフォルダ** に対応させる。

想定構造:

- `c01`
- `c02`
- `c03`
- ...

各コマフォルダには少なくとも以下を持たせる。

- コマ枠線用ラスタ
- コマ内部の描画内容
- コマ内部に完全に収まるラスターレイヤー群

### 4.2 GP

GP はページ全体の master GP から対象コマ領域ごとに切り出し、  
**コマ内ラスターレイヤー** として出す。

### 4.3 フキダシ / テキスト

フキダシとテキストは B-Name 内で別データなので、出力でも分けられる。

推奨:

- フキダシ本体: `balloon`
- 文字: `text`

として別レイヤー化する。

ただし、将来の実装を簡単にするなら初期版は

- `balloon_and_text`

の 1 枚にまとめてもよい。

### 4.4 効果線

効果線は現在 Grease Pencil ストロークへ落としているため、  
初期版では GP と同じパイプラインでラスタ化する。

推奨:

- `effect_lines`

として GP 本体とは別レイヤーに分ける。

## 5. コマ内 / ページ直下の振り分け規則

将来再開する場合、ラスタ化対象は次の規則で振り分ける。

### 5.1 コマフォルダ内に入れるもの

- その要素の表示領域が **単一コマの内側に収まる**
- コママスクで切れて問題ない

対象候補:

- GP
- 効果線
- フキダシ
- フキダシ内テキスト

### 5.2 ページ直下に置くもの

- コマをまたぐ
- 裁ち落としまで伸びる
- コママスクで切れては困る

対象候補:

- 見開きまたぎの効果線
- コマ外にはみ出すフキダシ
- ナレーション
- 複数コマに跨る擬音

## 6. 再開時の推奨実装順

`.clip` へ直接突っ込まず、次の順で進める。

### Phase A: B-Name 内部のラスタ化基盤を作る

先に B-Name のエクスポート基盤を拡張し、以下を Pillow 画像へ合成できる状態にする。

- コマレンダー
- GP
- フキダシ
- テキスト
- 効果線

この段階で必要なこと:

- ページ全体のラスタ書き出し
- コマ単位のラスタ書き出し
- どの要素がどのコマに属するかの判定
- ページ直下レイヤーに回す条件整理

### Phase B: まず PSD/PSB で階層出力する

`.clip` 前の中間ゴールとして、PSD/PSB で次を実現する。

- コマごとのレイヤーフォルダ
- コマ内ラスターレイヤー
- ページ直下ラスターレイヤー

これが通れば、B-Name 側のレイヤー分解ロジックはほぼ固まる。

### Phase C: `tools/clip_probe/` で最小 `.clip` 生成を検証する

アドオン本体とは分離し、別スパイクとして進める。

目標:

- 空の `.clip` を生成して CSP で開ける
- 1 個のコマフォルダだけを持つ `.clip` を生成して開ける
- コマ枠ラスタが正しく表示されることを確認する

### Phase D: B-Name 本体へ統合する

最後に B-Name の UI / export operator へ接続する。

## 7. 再開時の具体タスク

### 7.1 B-Name 側

- `io/export_pipeline.py` に GP 合成を追加
- フキダシの Pillow 描画を追加
- テキストの Pillow 描画を追加
- 効果線のラスタ合成を追加
- コマ内 / ページ直下の振り分けロジックを追加
- コマ単位 RGBA 画像生成 API を追加

### 7.2 `.clip` 側

- CSFCHUNK ラッパー生成
- `CHNKSQLi` の最小 SQLite 生成
- `ExternalChunk` と `CHNKExta` の参照整合
- コマフォルダ用 `Layer` 行生成
- コマ枠ラスタ用 `Mipmap` / `Offscreen` / 外部 chunk 生成
- コマ内ラスターレイヤー生成
- ページ直下ラスターレイヤー生成

## 8. 既知の注意点

### 8.1 破線・点線・二重線

B-Name のフキダシには `line_style` があるが、再開時点で見た目の完全再現ができるとは限らない。  
初期版はまず **実線化してでも出す** 方針を許容する。

### 8.2 ベタフラの塗り

効果線の `beta_flash` は、将来ラスタ化時に塗りを明示的に焼き込む必要がある。  
GP ストロークだけでは塗りの見た目が揃わない可能性がある。

### 8.3 テキスト組版の完全一致

縦書き、ルビ、縦中横、白フチまで B-Name 側の組版結果をラスタ化する必要がある。  
CSP 側で編集可能なテキストに戻すことは初期目標に含めない。

## 9. 再開時の受け入れ条件

再開時の最低限の完成条件は以下。

- CSP で `.clip` を開いて落ちない
- コマがコマフォルダとして見える
- コマ内の GP がラスタで見える
- フキダシ / テキスト / 効果線が意図した位置で見える
- コマをまたぐ要素が意図せずクリップされない

## 11. 判断メモ

この計画の本質は、

- **CSP のネイティブ編集性を無理に再現しない**
- **見た目が崩れず、作業を引き継げる `.clip` を作る**

ことにある。

したがって再開時も、優先順位は次の順に固定する。

1. 開けること
2. 見た目が合うこと
3. レイヤー構造が実用的であること
4. CSP ネイティブ編集性
