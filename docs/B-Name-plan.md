# B-Name（Blender ネーム作画アドオン）開発計画書

**最終更新**: 2026-04-26
**反映内容**: 2026-04-26 時点のユーザー指示と現行実装（用紙設定、枠線ツール、コマ編集モード、PSD 書き出し）を反映

## 1. 作品の概要

### 1.1 目的
Blender上で動作する、漫画ネーム作画専用のアドオン。3D操作と2D作画（コマ/フキダシ/ペイント）をシームレスに統合し、CLIP STUDIO PAINTの3D機能と同等以上の体験を、Blenderの3Dエンジンの上で実現する。

### 1.2 立ち位置
- **Meldex** — ストーリー構成・シナリオ管理・台本・DB管理を担う統合ワークスペース
- **B-Name** — Blenderアドオン。Meldexから送られたシナリオを元に、実際のネーム作画を行う
- **CLIP STUDIO PAINT** — 完成原稿の仕上げを担う（B-Nameでは原稿仕上げまでは行わない）

### 1.3 開発ディレクトリ
`D:\Develop\Blender\B-Name\`

### 1.4 Meldexとの関係
**独立したアドオンとして開発する。** Meldexとの連携は一方向のみ:

- Meldex → B-Name: 「クリスタへ送信」と同様の「B-Nameへ送信」機能。シナリオ（台本テキスト＋ページ構成）を送信
- B-Name → Meldex: 連携なし（B-Name内で完結）

Meldexのソースコードには一切依存しない。Meldex側に小さな送信機能を追加するのみ。

#### 1.5.2 コマ編集モードの現行仕様
- 現行コードでは、コマ編集は **1コマ = 1 `cNN.blend`** の独立ファイル方式で遷移する
- `cNN.blend` の新規作成は `work.blend` の単純コピーではなく、空 scene から初期化する方向へ切り替わっている
- `cNN.blend` へ入る直前に `work.json` / `pages.json` / `page.json` を保存し、用紙色 / 開始ページ / 読む方向 が `load_post` で古い JSON に巻き戻らないようにする
- コマ編集モード中も、同じページの他コマと他ページを読取専用の参照オーバーレイとして表示する
- 参照表示は構図合わせ用であり、直接編集対象は現在開いている `cNN.blend` のコマ内容に限定する

#### 1.5.3 コマプレビュー反映の現行仕様
- 現行コードでは、コマ編集モード終了時に `cNN_thumb.png` を自動更新する経路がある
- 任意で `cNN_preview.png` を生成する経路もある
- ページビューポート側と書き出し側は、`cNN_preview.png` / `cNN_thumb.png` のうち更新日時が新しい方を参照する
- コマ編集モード終了時の自動スクショでは参照オーバーレイを一時的に抑制し、他ページの参照表示がコマサムネイルに混入しないようにする

#### 1.5.4 枠線ツールの現行仕様
- `水平カット` / `垂直カット` の個別ボタンは廃止
- `縦横均等分割` は枠線ツール文脈へ移動し、選択中コマを対象に実行する
- 対象コマは枠線選択ツールの選択コマを優先し、矩形だけでなく polygon にも対応する
- 枠線選択ツールでは、枠線全体 / 1辺 / 1頂点単位で線色・線幅を編集できる
- `コマ:枠線` セクションは、アクティブコマではなく枠線選択ツールの選択対象を優先する
- 枠線カットツール / 枠線選択ツールのモーダル中でも、B-Name パネルとビューポート右上ナビゲーションは操作可能とする
- `▲` 拡張は選択辺の角度を保ったまま平行移動で候補へ重ね、自己交差・面積ゼロ・頂点順序反転を起こす案は採用しない

#### 1.5.5 用紙設定の現行仕様
- 用紙パネルから `用紙色` を直接編集できる
- 開始ページの既定値は `左から`
- セーフライン外側オーバーレイの新規作品既定色は、明度 `0.7`

#### 1.5.6 書き出しの現行仕様
- 実用優先の主目標は PSD レイヤー出力
- 現行コードでは `psd-tools` 前提で PSD 書き出し経路を持ち、各要素の個別レイヤー化を進めている
- コマは **コマ形状のレイヤーマスクが掛かったフォルダ** として扱う方針を採る
- `.clip` 直書きは現時点では着手しない
  - 調査結果と延期理由は `docs/clip_export_deferred_plan.md` に残す

## 2. 採用技術の根拠

### 2.1 なぜBlenderアドオンか
- Blenderビューポートをブラウザ（Meldex）に埋め込む公式手段は存在しない
- 3D操作と2D作画のシームレス統合には、Blenderの描画スタックへの直接アクセスが必須
- Grease Pencilオブジェクトは3D空間内で2D描画を行う既存機能であり、ネーム作画の基盤として最適

### 2.2 Pythonの速度・UI懸念への対応
- **速度**: 描画処理本体はBlenderのC++エンジンが担当。Pythonの負荷は「頂点ループ」のみ。ネーム作画では致命的にならない
- **UI幅**: Blender標準UI（Panel / Operator / Popup / Modal）で必要十分。カスタムUI が必要な箇所は `gpu` モジュールでオーバーレイ描画
- **不足分の対処**: どうしてもPythonで重い処理はC拡張化を後段で検討（ただしMVPでは不要）

## 3. 機能要件

### 3.1 基本作画機能

#### 3.1.1 レイヤー機能
2種類のレイヤーを扱う:

- **ベクターレイヤー**（Grease Pencil v3ベース） — 線画・フキダシ・塗り
- **画像レイヤー**（ビットマップ） — スキャンラフ取り込み・写真参照・実写背景

共通機能:
- レイヤー追加・削除・複製・並び替え・結合
- 表示/非表示・ロック・不透明度・ブレンドモード
- レイヤーフォルダ — **Grease Pencil v3 の `GreasePencilLayerGroup` をそのまま使用**（独自実装不要）

Grease Pencil v3 API前提:
- データブロック: `bpy.data.grease_pencils_v3` / `bpy.types.GreasePencilv3`
- レイヤー: `bpy.types.GreasePencilLayer`（v3版、v2の`GPencilLayer`とは別物）
- レイヤーグループ: `bpy.types.GreasePencilLayerGroup`
- 描画: `bpy.types.GreasePencilDrawing`
- ストロークは属性（`curves` attribute API）ベース。大量生成時は一括APIを使う

画像レイヤー固有機能:
- 画像ファイル読み込み（PNG/JPG/TIFF/PSD）
- **描画方式: `SpaceView3D.draw_handler_add` + `gpu` モジュールでオーバーレイ描画**（3.4.3aと統一）
  - 平面メッシュ方式は採用しない（シェーディングモード依存・Zソート破綻の懸念）
  - Grease Pencil v3 の画像リファレンス機能も不採用（Blender側仕様で制約あり）
- 位置・拡大縮小・回転・反転
- 不透明度・ブレンドモード（`gpu.state.blend_set` で制御）
- 簡易レベル補正（明度/コントラスト/2値化）— 下書き取り込み用途
- 書き出し時はPillowで合成再描画（3.8 書き出しパイプライン参照）

#### 3.1.2 ペイント機能
- Grease Pencilブラシをラッピング
- ペン/鉛筆/マーカー/消しゴム/塗りつぶし
- ブラシサイズ・不透明度・筆圧
- カラーピッカー（Blender標準 ColorRamp を活用）
- 定規・直線・図形ツール

#### 3.1.3 コマ割り機能
- コマ = 平面メッシュ or Grease Pencilレイヤーセット
- コマ枠の自動生成（縦分割/横分割/カスタム）
- コマ枠のドラッグ編集
- コマごとにローカル座標系を持ち、その中で3Dモデル配置・作画が可能
- コマ境界でのクリッピング

#### 3.1.4 フキダシ・テキスト機能

##### 3.1.4.1 構造
- フキダシ = **本体シェイプ**（ベジェ曲線 or Grease Pencilストローク）+ **尻尾**（形状により直線/曲線/付箋型）+ **テキストレンダラ**
- テキストは 3.1.5 の独自組版エンジンでレンダリング（縦書き・ルビ・縦中横対応）
- フキダシ本体とテキストは独立編集可能（本体を変形してもテキスト配置は自動追従）

##### 3.1.4.2 フキダシ形状（Meldexボードのカード形状と同一）
以下の5種類をフキダシ本体の形状プリセットとして同梱。Meldexのボードアプリのカード形状と**見た目・パラメータ一致**で実装する:

| 形状 | 説明 |
|---|---|
| **矩形** | 直線で囲まれた長方形 |
| **楕円** | 楕円形のフキダシ |
| **雲** | モクモクとした雲形（波の数・振幅を調整可）。思考向け |
| **トゲ（曲線）** | 外周がギザギザで各トゲが曲線を描く（叫び・強調向け）。トゲ数・深さ・ばらつきを調整可 |
| **トゲ（直線）** | 外周がギザギザで各トゲが直線（叫び・強調向け）。トゲ数・深さ・ばらつきを調整可 |

それぞれに対して:
- 本体サイズのリサイズ（幅/高さ）
- 形状固有パラメータ（雲の波数、トゲ数・深さ等）
- 回転
- 線種・線幅・線色
- 塗り色（既定: 白／透明）
- **角丸オプション**（3.1.4.2a 参照）
- **尻尾の追加**（3.1.4.5 参照、複数可）

##### 3.1.4.2a 角丸オプション（全形状共通）
**角丸はフキダシ形状ではなく、全形状に共通で適用できるオプション。**

- ON/OFF
- 角半径（mm）
- 矩形では四隅に適用。雲・トゲの各ピーク/谷の角にも適用可能（トゲが柔らかくなる）
- 楕円のように既に丸い形状では無効（UI上はグレーアウト）

##### 3.1.4.2b カスタムフキダシ形状（パス登録）
ユーザーがパスツールで作成した任意の形状を、**フキダシ形状プリセットとして登録**できる。

機能:
- **パスから登録**: 選択中の閉じたパス（ベジェ曲線 / Grease Pencilストローク / 多角形）を「フキダシ形状として登録」メニューから登録
- **名前を付けて保存**: 登録時にプリセット名と説明を入力
- **縮尺・アスペクト比の扱い**:
  - 絶対座標ベース（登録時のサイズのまま使用）
  - 相対座標ベース（フキダシ本体のサイズに合わせてスケーリング）
  - 登録時に選択可能
- **編集可能属性**: 登録後もプリセットの頂点を再編集可能
- **削除・リネーム**: プリセット一覧からの管理

保存場所（2層構造、3.4.4のアセットライブラリ2層構造と整合）:
- **作品ローカル**: `MyWork.bname/assets/balloons/` に `.json`（パスデータ）として保存。作品内で使える
- **グローバル**: アドオン設定から「グローバルプリセットとして登録」を選ぶと `<Blenderユーザー設定>/scripts/addons/B-Name/presets/balloons/` に保存。全作品から使える

`work.json` の `shapeMapping` からカスタムプリセット名を参照することで、セリフ種別→カスタム形状のマッピングも可能。

既定の5形状と同様に、カスタム形状にも以下が適用される:
- 角丸オプション（3.1.4.2a）
- 尻尾の追加（3.1.4.5）
- 線種・線幅・線色・塗り色

##### 3.1.4.3 フキダシなしテキスト
- **フキダシ本体を持たないテキスト単体オブジェクト**も作成可能
- 用途: 擬音／手書き風の台詞／ナレーション／心の声／場面転換の文字等
- 配置・移動・回転・サイズ変更はフキダシと同じUIで操作
- テキスト属性（フォント・サイズ・白フチ等）はフキダシ内テキストと共通

##### 3.1.4.4 テキスト属性
- フォント（作品既定／個別指定）
- フォントサイズ（pt）
- 行間・字間
- 縦書き／横書き切り替え
- ルビ・縦中横
- 文字色
- **白フチ（テキストストローク）** — 指定幅（mm）でテキスト外周を白（または指定色）で縁取り
  - 用途: 背景の線画やトーンの上にテキストを乗せたとき視認性を確保
  - フキダシなしテキストでは特に重要
  - 幅（既定 0.2mm）／色（既定 白）／ON/OFF
  - フキダシ内テキストでも個別にON可能

##### 3.1.4.5 尻尾（しっぽ）
**尻尾はすべてのフキダシ形状に共通で追加できるオプション。** Meldexの注釈ツールの付箋タイプを含む以下から選択:

- **直線尻尾** — 三角形の直線状しっぽ
- **曲線尻尾** — ベジェで膨らませた曲線状しっぽ
- **付箋型しっぽ** — Meldex注釈ツールの付箋のような矩形タブ状のしっぽ

各尻尾に対して:
- 方向
- 長さ
- 根元幅・先端幅
- 曲線尻尾ではカーブの曲がり具合

1フキダシに複数尻尾を持てる（話者共有セリフ・ハモり等）。フキダシなしテキストにも尻尾を追加可能（擬音から線を引き出す等）。

##### 3.1.4.6 セリフ種別と形状のマッピング
Meldexから受信したセリフ種別を、既定のフキダシ形状（+角丸オプション）にマッピング:

- 通常セリフ → 矩形（角丸ON）
- 思考 → 雲
- 叫び → トゲ（直線）
- ナレーション → 矩形（角丸OFF）
- カスタム → ユーザー設定

マッピングは作品単位で変更可能（`work.json`）。

#### 3.1.5 日本語組版（縦書き）エンジン
Blender標準テキストオブジェクトは縦書き・ルビ・縦中横に対応していないため、**独自組版エンジンを実装する。**

必須機能:
- **縦書き** — 右→左の行進行、文字の縦配置、句読点・括弧の回転/位置調整（約物処理）
- **ルビ** — 親文字に対してモノルビ/グループルビ/熟語ルビ。親文字との中心合わせ、ルビ文字サイズ比率設定
- **縦中横** — 半角数字・英字の連続を縦書き内で横向きに表示（「25」「！？」等）
- **行間（行送り）** — 行ピッチの数値設定
- **字間（字送り）** — 文字ピッチの数値設定、詰め組み対応
- **禁則処理** — 行頭禁則（、。」等）、行末禁則（「等）、ぶら下げ
- **字形選択** — 縦書き用グリフ（縦組み用カギ括弧、ダッシュ等）への自動切り替え

実装方針（3層構成）:

| レイヤー | モジュール | 役割 |
|---|---|---|
| ビューポート表示 | `blf`（Blender標準） | 画面上の描画。`blf.size(fontid, size)` の現行シグネチャ前提 |
| 書き出し再描画 | `Pillow` | 書き出し画像に合成描画（`ImageDraw.text` 等）。`blf`の結果は書き出し画像には焼き込まれないため、同じロジックで再描画する必要がある |
| グリフ選択・メトリクス | `fontTools` | OpenType `vert` フィーチャによる縦書き字形切替／字送り計算／ルビ・縦中横メトリクス。ビューポート側・書き出し側の両方で共用 |

その他:
- Blenderテキストオブジェクトは使わない（縦書き・ルビ非対応のため）
- Grease Pencilストロークとしての文字配置は採用しない（v3でのストローク属性APIが重い / 書き出し整合性が悪い）
- フォント選択はユーザー設定（OTF/TTF直接指定）
- Meldexからのルビ情報（親文字＋ルビ文字）を構造データのまま保持
- 関連モジュール: `typography/viewport_renderer.py`（blf）、`typography/export_renderer.py`（Pillow）、`typography/metrics.py`（fontTools、両者で共用）

#### 3.1.6 効果線ツール（集中線・ベタフラ・ウニフラ・流線）
CLIP STUDIO PAINTの「ウニフラ」「集中線」「流線」相当の効果線を生成する専用ツール。ドラッグで範囲指定→パラメータ調整で即座に効果線レイヤーを生成する。

##### 3.1.6.1 効果線の種類
| 種類 | 説明 |
|---|---|
| **集中線** | 長方形・楕円・多角形の基準図形から外側（または内側）へ放射状に線を描く |
| **ウニフラ**（ウニフラッシュ） | 集中線の一種で、基準図形がギザギザ（ウニ状）になっているタイプ |
| **ベタフラ**（ベタフラッシュ） | ウニフラの塗りつぶし版。外周がギザギザの黒ベタ領域 |
| **流線** | 直線または曲線に沿って平行な線を複数並べる動き・速度表現 |

##### 3.1.6.2 基準図形（集中線・ウニフラ・ベタフラ共通）
- **長方形** / **楕円** / **多角形**（頂点数指定）
- **放射線定規を中心に用いる**（事前に定規を置いた場合）
- **中央から開始**: 基準図形の中心から線を開始するか、外周から開始するか
- **縦横指定**: 比率指定 または 長さ指定
- **確定後の角度調整**: 効果線全体の回転（角度刻み指定可）

##### 3.1.6.3 線のパラメータ
- **ブラシサイズ**（mm、既定 0.40mm）
- **乱れ**（ブラシサイズのゆらぎ、ON/OFF、量）
- **線の間隔**: 角度指定 または 距離指定
  - 距離指定の場合の間隔値（mm、既定 0.40mm）
  - 間隔の乱れ（ON/OFF、量）
- **まとまり**（線を束にする機能）:
  - 束の乱れ（ON/OFF、量）
  - 束内の隙間
- **長さ**（mm、既定 10.0mm）
  - 長さの乱れ（ON/OFF、量）
  - **コマ外まで延長**（ON/OFF、コマ境界を越えて延長するか）
- **基準位置**: 線の起点（始点／中点／終点）
  - 基準位置のずれ（既定 2.0）
  - **基準位置をギザギザにする**（ウニフラ・ベタフラで利用、数・高さを指定）
- **入り抜き**:
  - 適用モード（ブラシサイズ／長さ／不透明度 等）
  - 入り（0-100%、既定100）
  - 抜き（0-100%、既定100）

### 3.2.5 コマ枠機能

#### 3.2.5.1 コマ枠の基本属性
各コマ枠は以下の属性を持つ:

- **形状**: 長方形／多角形／曲線（ベジェ）／フリーフォーム
- **Z順序**: 重なり順を決める整数値（後述 3.2.5.5）
- **枠線スタイル**:
  - 線種: 実線／破線／点線／二重線／カスタムパターン
  - 線幅（mm）
  - 線色
  - 角処理: 直角／丸角（半径指定）／面取り
  - **辺ごとの個別オーバーライド**（上辺だけ太線、右辺だけ破線、等）
  - **枠線全体の非表示**（コマの機能は維持したまま線のみ非表示）
- **白フチ**:
  - ON/OFF
  - 幅（mm）、**既定 0.37mm**
  - 色（既定: 白）
  - 枠線の**外側**に描画
  - 辺ごとの個別オーバーライド
- **対応するコマ3Dシーン**: `cNN.blend` のパス（4章参照）
- **作画レイヤー参照**: このコマに紐づくGrease Pencilレイヤー・画像レイヤー・フキダシのIDリスト

#### 3.2.5.2 コマ枠の作成・編集ツール
- **矩形コマ作成**（ドラッグで矩形範囲指定）
- **縦横均等分割**:
  - UI は独立した「分割テンプレート」セクションではなく、**枠線ツール**内に置く
  - 対象はページ全体ではなく、**枠線選択ツールで選択中のコマ**
  - **矩形コマだけでなく polygon コマにも対応**する
- **枠線カットツール**（CLIP STUDIO PAINTの同名ツール相当）:
  - 既存のコマをドラッグで切断
  - 切断線は直線／自由線／多角線
  - 切断後、両側のコマに作品内共通のコマ間隔（スキマ）が自動適用
- 水平カット／垂直カットの個別ボタンは置かない
- **枠線選択ツール**:
  - 辺単位／頂点単位／枠線全体単位で選択する
  - 色・線幅はコマ全体だけでなく、**1 辺だけ／1 頂点に接続する 2 辺だけ**にも適用できる
  - 選択辺の両端頂点はハイライト表示する
  - ハンドルクリック1回、ドラッグ確定1回をそれぞれ独立した undo step として扱う
  - ツール使用中でも B-Name パネルとビューポート右上ナビゲーションは操作できる
- **頂点編集**: 長方形を多角形化、頂点追加・削除・移動、曲線化（ベジェ化）
- **結合**: 複数コマを1つに結合
- **再分割**: 結合済みコマの再分割
- **フリーフォーム描画**: Grease Pencilストロークを閉じてコマ化

#### 3.2.5.3 スナップ機能
コマ枠の作成・移動・リサイズ時に以下へスナップする:

- **仕上がり枠**（裁ち切り線）
- **基本枠**（内枠・本文領域）
- **セーフライン**（天・地・ノド・小口）
- **トンボ**
- **他のコマ枠**（作品共通のコマ間隔を保ってスナップ）

Shiftキー押下でスナップを一時無効化。スナップのON/OFFはアドオン設定で切替可能。

#### 3.2.5.4 コマ間隔ルール（作品内共通）
作品単位で以下を保持し、スナップ・枠線カット後の自動マージンに適用する:

- **上下スキマ（縦の間隔）**: mm単位、**既定 7.3mm**
- **左右スキマ（横の間隔）**: mm単位、**既定 2.1mm**
- これらの値は `work.json`（作品メタデータ）に保存され、作品内の全ページ・全コマで共有
- コマ単位で個別オーバーライド可能（通常コマ隣接時はルール優先、ぶち抜きや変形コマでは個別値を許容）
- 新規作品作成時の初期値はアドオン設定で変更可能

#### 3.2.5.5 Z順序・コマの重なり
- 同一ページ内のコマ枠同士が重なる配置を許容（飛び出しコマ／挿入コマ／アイキャッチ等）
- 各コマは整数のZ順序値を持つ
- **重なり順変更機能**:
  - コマ右クリック → 「最前面へ」「最背面へ」「1つ前面へ」「1つ背面へ」
  - コマ一覧パネルでのドラッグ&ドロップ並び替え（Z順序に連動）
- **自動くり抜き**（既定ON）:
  - 上位Zのコマが重なる範囲は、下位Zのコマから自動的にくり抜かれる
  - 下位コマのレンダリング範囲・作画レイヤークリッピングに反映
- コマ単位で「くり抜かない」設定にも切替可能
- **ぶち抜きコマ**: 仕上がり枠を超えて裁ち切りまで広がるコマも同じ仕組みで扱う

### 3.2.6 セーフライン外側オーバーレイ

#### 3.2.6.1 機能概要
セーフラインの外側（本文として使えない範囲）を指定色で塗りつぶし、乗算レイヤーとして全体に重ねて表示する。作画中に「裁ち切り・綴じで見えなくなる領域」を視覚的に区別する。

#### 3.2.6.2 仕様
- ON/OFF切り替え（ツールバー / 用紙設定パネル）
- 対象範囲: セーフラインの外側（未設定時は基本枠の外側にフォールバック）
- 塗りつぶし色: 既定 **明度 0.7 のグレー**（`#B3B3B3` 相当、ユーザー変更可）
- ブレンドモード: 乗算（既定）／通常／その他
- 不透明度: 既定 30%（0〜100%スライダー）
- **表示専用レイヤー** — 書き出し結果には一切含まれない（ビューポート上のガイドのみ）
- セーフライン内側の矩形はくり抜いて透過
- 見開きページではノド位置を考慮し左右それぞれのセーフラインでくり抜き
- 作品共通既定（`work.json`）、ページ単位でオーバーライド可

#### 3.3.1 ページ管理

- ページ追加・削除・並び替え・複製
- 見開き表示切り替え
- **ページを見開きに変更・見開きを解除**（詳細は 3.3.4）
- ページ間ジャンプ
- ページ順序・メタデータは `pages.json` で一元管理（4章参照）

#### 3.3.2 コマ管理
- コマ一覧パネル（ページ内のコマをサムネイル表示）
- コマ追加・削除・複製
- **コマを他のページへ移動**（ドラッグ&ドロップ / 右クリックメニュー）
- **コマを他のページへ複製**（同上）
- コマ=独立ファイル方式のため、移動・複製はファイル単位で完結（4章参照）
- コマ内の作画レイヤー（Grease Pencil・フキダシ）も同時に移動・複製される

#### 3.3.3 コマ移動時の内部処理
1. 移動元ページの `cXX/` ディレクトリ（`cXX.blend` / `cXX.json` / `cXX_thumb.png`、存在すれば `cXX_preview.png` や `passes/` も含む）を移動先ページ直下へ移動
2. 移動先でファイル名重複があれば自動リネーム（空き番号へ）
3. 移動元 `page.json` からコマエントリ削除、移動先 `page.json` に追加
4. 紙面シーン（`work.blend`）の更新:
   - 移動元: コマ枠オブジェクトと紐づく作画レイヤー・フキダシを削除（またはエクスポート）
   - 移動先: コマ枠オブジェクトと作画レイヤー・フキダシを配置
5. レンダーキャッシュ画像はそのまま持ち越し（再レンダー不要）

#### 3.3.4 ページの見開き変更・解除
CLIP STUDIO PAINTの「見開きに変更」相当の機能。

##### 操作UI
- ページ一覧パネルで隣接する2ページを選択 → 右クリック → 「見開きに変更」
- 見開き変更ダイアログ:
  - 「NページとMページを見開きとして結合します」の情報表示
  - 「対象ページを開いている場合は編集中の内容でファイルに保存されます。この動作は[取り消し]を実行できません。」の警告
  - **トンボを合わせる**（ON/OFF、既定ON）
    - 間隔（mm、既定 `-9.60mm`、負値はページを重ねる方向）
  - **描画されていないレイヤーを削除**（ON/OFF、既定ON）
  - OK / キャンセル

##### 内部処理
1. 対象2ページの編集内容を自動保存（Undo不可の旨を警告表示）
2. 新しい見開きディレクトリを作成: `pNNNN-MMMM/`
3. `work.blend` / `pages.json` / 各 `page.json` を更新し、見開き用のページエントリと表示配置を再構築する（トンボ間隔を考慮）
4. コマディレクトリは元ページ直下から新見開きページ直下へ移動（ID衝突時は自動リネーム）
5. 「描画されていないレイヤーを削除」がONの場合、空のGrease Pencilレイヤー・画像レイヤーを削除
6. `pages.json` の該当2エントリを1つの `spread: true` エントリに置換、`originalPages` に元ページIDを保持
7. トンボ合わせの間隔（mm）を `page.json` に記録

##### 見開き解除
- 見開きページを右クリック → 「見開きを解除」
- `originalPages` の情報に基づき、2ページに分割して元のディレクトリ構造へ戻す
- 結合後に追加された作画は、ページ境界を基準に左右どちらかへ自動振り分け（境界上にまたがるものは警告）

### 3.4 3D連携機能（シームレス統合の本丸）

#### 3.4.1 2層構造（紙面ワールド / 3Dワールド）
- **紙面シーン**: 原稿用紙・コマ枠・Grease Pencilレイヤー・画像レイヤー・フキダシを保持。ビュー用カメラ（3.6章のカメラリグ）がこのシーンを撮影
- **コマごとの3Dシーン**: 各コマが独自の3D空間を持つ。背景モデル・キャラクター・ライティング・シーンカメラを保持
- 同一ページ内の異なるコマで**まったく別の場所・別の登場人物**を扱える
- 紙面シーンのコマ枠は、対応するコマファイルのレンダー結果をテクスチャとして貼り込んで表示

#### 3.4.2 独立ファイル方式（1コマ=1.blendファイル）
- 各コマの3Dシーンは独立した `cXX.blend` ファイルとして保存
- 利点: メモリ効率 / コマの移動・複製が容易 / バージョン管理との相性 / 破損の局所化
- コマファイルは3Dモデル本体を持たず、**アセットライブラリからのリンク参照のみ**で構成
- 紙面全体の正本は `work.blend` と JSON 群で持ち、`cXX.blend` はそのコマの3D編集内容を保持する
- コマ編集モードへ入っても、**用紙色 / 開始ページ / 読む方向など作品共通設定は `work.json` と一致**し続ける

#### 3.4.3a コマ枠画像・画像レイヤーの描画方式（全体方針）
B-Nameでビューポート上に表示する以下の要素は、すべて**`SpaceView3D.draw_handler_add` + `gpu` モジュールによるオーバーレイ描画**で統一する:

- コマ枠内に貼り込むサムネイル/プレビュー画像
- 画像レイヤー
- セーフライン外側オーバーレイ
- コマ枠線・白フチ
- テキスト白フチ（`blf` と併用）
- 原稿上の作品情報・ノンブル

理由:
- Blenderのシェーディングモード（Solid / Material Preview / Rendered）に依存せず描画できる
- Zオーダー・重なりくり抜きを自前で管理でき、3.2.5.5の仕様と整合
- 新規コマ時の「背景色白のみ」フォールバックが素直に実装できる（画像未生成なら単純に描画スキップ）
- 書き出し時は同じ描画ロジックをPillowで再実装（共通関数化）して画像合成

**書き出し時は `draw_handler` の出力がレンダリング画像に焼き込まれないため、Pillowで再描画する（詳細は 3.8 書き出しパイプライン）。**

シェーダーは `gpu.shader.from_builtin` を使い、ビルトイン名は最新系（例: `UNIFORM_COLOR` / `IMAGE` 等、Blender 4.0以降の新名）。

#### 3.4.3 スクショサムネイル方式（コマ外での表示）
**コマ編集モードを閉じるたびにレンダリングは行わない。** レンダリングは書き出し時のみ実行する。コマ枠に表示する画像は以下の方針で扱う。

| 状況 | コマ枠に表示されるもの |
|---|---|
| コマ編集モード中 | Blenderビューポートのライブ描画（そのまま） |
| 編集モード終了時 | **ビューポートのスクリーンショットを取得し `cNN_thumb.png` に保存**（レンダリングではない、数十ms程度） |
| 過去にスクショがある | そのスクショを表示 |
| **スクショがまだない（新規コマ作成直後など）** | **コマ枠の背景色（白）のみ表示**（プレースホルダ画像は生成しない） |
| ユーザーが明示的に「高品質プレビュー生成」を要求した時 | そのときだけ低解像度レンダリングを実行し `cNN_preview.png` に保存 |
| 書き出し | 書き出し時のみ高解像度で再レンダリング（3.8章） |

- 紙面編集モードへ戻った時は、**直前に編集したコマの内容がページ一覧・紙面上に即反映される**こと
- `cNN_preview.png` と `cNN_thumb.png` の両方がある場合は、更新日時が新しい方を使う。古い高品質プレビューが編集後の自動サムネイルを隠さないことを優先する
- コマ編集モードは「そのコマ以外が一切見えない空シーン」にはせず、**同一ページの他コマと前後ページを参照表示できる**ようにする

スクショ取得の実装:
- 第1候補: **`bpy.ops.screen.screenshot_area()`**（`temp_override` で正しいarea/region/windowを渡す）
- フォールバック: `gpu.types.GPUOffScreen` + `GPUOffScreen.draw_view3d()` によるオフスクリーンキャプチャ
  - Vulkan/Metal/OpenGL の各バックエンドで動作検証が必要
  - `gpu.state` で `blend_set` / `depth_test_set` を明示
- **解像度はコマ枠の表示サイズに合わせた最小限**（画面上の表示サイズの等倍〜2倍程度）
- レンダリング処理を一切通らないため、ほぼ瞬時に完了
- 紙面表示側は画像ファイルの更新日時を見て、既存の `bpy.data.images` キャッシュを必要時に reload する

手動「高品質プレビュー」:
- コマ右クリック → 「高品質プレビューを生成」
- そのコマだけ低解像度レンダリング（Freestyle線画化・ライティング含む）を実行
- 結果は `cNN_preview.png` に保存
- スクショ（`_thumb.png`）とプレビュー（`_preview.png`）は共存。表示切替スイッチあり（既定: 更新日時が新しい方を優先）

書き出し時との差異:
- スクショ・低解像度プレビューと、書き出し結果（高解像度レンダリング）では見た目が多少異なる
- Freestyle線画・ライティング最終計算等は書き出し時に最終確定される
- これは仕様として許容

#### 3.4.4 3Dアセット管理（2層のアセットライブラリ）
すべてBlender標準のアセットブラウザから利用:

1. **グローバルアセットライブラリ** — 全作品共通
   推奨パス例: `D:\Develop\Blender\B-Name-Assets\`
2. **作品固有アセット** — その作品専用
   `MyWork.bname/assets/`

実装方針:
- **B-Name独自のアセット管理UIは作らない** — Blender標準のアセットブラウザをそのまま活用
- アセットブラウザからコマ編集モード中のシーンへ、ドラッグ&ドロップでリンク追加
- アドオン設定画面に「アセットライブラリ登録ガイド」を表示（ユーザーが自分でBlender設定に登録）

#### 3.4.5 モデル本体の編集
- 3Dモデル本体の編集は**Blender本体で直接 `.blend` を開いて行う前提**
- B-Name上ではリンク参照したモデルのアセンブル（配置・ポーズ）のみ行う
- **カスタム右クリックメニュー**: `右クリック > B-Name > リンク元ファイルを開く`
  - デフォルト動作: 新しいBlenderインスタンスを `subprocess` で起動して開く（現在の編集作業を失わない）
  - 設定で「現在のBlenderで開く」に切替可能

#### 3.4.6 その他の3D機能
- 3Dモデルのポーズ編集（コマ単位で独立）
- 3Dモデルの線画化（Freestyle / Grease Pencil変換）
- 3D→2D投影して作画レイヤーへ落とし込み

**※ パース定規機能は実装対象外**（3Dカメラがパース情報を直接持つため不要）

### 3.5 Meldex連携（受信のみ）
- Meldexから「B-Nameへ送信」でシナリオデータを受信
- 受信データ構造（JSON想定）:
  - ページ構成（ページ数・見開き情報）
  - 各ページのコマ数目安
  - コマごとのセリフ/ナレーション/ト書き
  - **セリフ内のルビ情報**（親文字・ルビ文字・範囲）
  - セリフ種別（通常/思考/叫び/ナレーション等、フキダシ種別に対応）
  - キャラクター情報（話者）
- 受信方法: ローカルHTTPサーバー or ファイル経由（後述の検討事項）
- 受信後、自動でページとフキダシの雛形を生成（ルビ情報も縦書きエンジンに渡す）

**Meldex側のデータ構造確認が必要** — ルビをMeldex内部でどう保持しているか（HTMLの `<ruby>` タグか、独自構造か）を調査した上で、送信フォーマットを策定する。

### 3.6 ビューポート操作ショートカット（CLIP STUDIO PAINT準拠）

CLIP STUDIO PAINTと同じ操作感でビューポートを扱えるよう、以下のショートカットを実装する。Blender標準のビューポート操作（中ボタンドラッグ等）とは別レイヤーで、B-Nameのアドオンが有効なビューポートでのみ動作するモーダルオペレータとして実装する。

#### 3.6.1 ショートカット一覧

| 操作 | ショートカット |
|---|---|
| 手のひら（パン） | `Space + ドラッグ` |
| 回転 | `Shift + Space + ドラッグ` / `Shift + マウスホイール` |
| ズームイン | `Ctrl + Space + クリック`（macOS: 先にSpace→Cmd） |
| ズームアウト | `Alt + Space + クリック` |
| ズーム（連続） | `Ctrl + Space + ドラッグ` / `マウスホイール` |
| レイヤー選択 | `Ctrl + Shift + クリック` または `Ctrl + Shift + ドラッグ` |
| スポイト | `右クリック` |

#### 3.6.2 実装方針
- Blender標準のキーマップとは独立した、B-Name専用キーマップを定義
- アドオン設定画面でショートカットをカスタマイズ可能にする
- Blender標準操作（中ボタンドラッグでのパン/回転、テンキーでの視点切り替え等）は従来通り有効のまま（併用可能）
- OS判定（Windows/macOS/Linux）はBlenderの `bpy.app` で行い、macOSでは `Cmd` キーを受け付ける
- スポイトは「右クリック=コンテキストメニュー」になっているBlenderの標準動作を上書きするため、アドオン有効時のみ切り替える

#### 3.6.3 モード依存の動作
- **紙面編集モード中**: すべてのショートカットは紙面シーンのビュー用カメラに作用
- **コマ編集モード中**: ショートカットはそのコマの3Dシーンのシーンカメラに作用
- **ペイント中**: すべてのショートカットが優先（ストローク描画より先に判定）
- **オブジェクト選択中**: レイヤー選択は作画レイヤーのみを対象、3Dオブジェクト選択はBlender標準キー（左クリック）を維持
- **テキスト編集中**（フキダシ内文字入力）: ビューポート操作ショートカットは一時無効化

#### 3.6.4 キー衝突の回避
- Blender既定の `Space` はツール切り替え/検索メニューに割り当てられているため、B-Name専用の操作モード時はこれを一時的にオーバーライド
- `Shift + マウスホイール` はBlender既定で水平スクロール等に使われる場合があるため、B-Nameビューポート上でのみアドオン側の処理を優先

#### 3.7.1 作品情報

- **作品名**
- **話数**
- **サブタイトル**
- **作者名**

#### 3.7.2 原稿上への表示

- 表示位置: `上／下` × `左／中央／右` の9通り（項目ごとに個別指定可）

### 3.8 書き出し機能

#### 3.8.1 単一ページ書き出し

- フォーマット: PNG / JPEG / TIFF / PSD / PDF
- カラーモード: モノクロ／グレースケール／RGB／CMYK

#### 3.8.2 複数ページ一括書き出し
CLIP STUDIO PAINTの「複数ページ書き出し > 一括書き出し」相当。

##### 対象ページ選択

##### 出力形式
- PNG / JPEG / TIFF / PSD / **PDF（全ページ結合）**
- カラーモード: モノクロ／グレースケール／RGB／CMYK

##### 連番ファイル名

##### 縮小出力
- **出力サイズ指定**:
  - 元解像度のまま
  - 倍率指定（50% / 25% / カスタム）
  - 最長辺ピクセル指定（例: 長辺1600px）
  - 解像度指定（dpi変更）
- **補間方法**: バイキュービック／ランチョス／バイリニア／ニアレストネイバー

##### 出力範囲
- 仕上がり枠まで
- 裁ち落とし込み
- 基本枠まで
- キャンバス全体

##### 追加オプション
- セーフライン外側オーバーレイは**含めない**（ビューポート専用のため常に除外）
- コマ枠線を**含める／含めない**
- 白フチを**含める／含めない**
- トンボを**含める／含めない**
- ノンブルを**含める／含めない**
- 作品情報を**含める／含めない**
- 用紙色を**含める／含めない**
- レンダーキャッシュを使用／強制再レンダリング

##### 出力先
- フォルダ選択
- 既定: `MyWork.bname/exports/<出力日時>/`
- 書き出し履歴を `work.json` に記録

##### 処理方式
- 進捗ダイアログ表示（「Nページ中Mページ処理中」）
- バックグラウンド処理でBlender UIをブロックしない
- キャンセル可能
- エラー発生ページはスキップして続行、最後にエラー一覧表示

#### 3.8.3 書き出しプリセット
- よく使う書き出し設定を名前付きで保存・呼び出し
- `work.json` の `exportPresets` に格納
- 既定プリセット（同梱）:
  - 「印刷入稿用PNG」（600dpi / 裁ち落とし込み / モノクロ）
  - 「確認用縮小PNG」（長辺1600px / RGB）
  - 「PDF結合（全ページ）」

#### 3.8.4 書き出しパイプライン（オーバーレイ要素の焼き込み）
Blenderの `draw_handler_add` によるオーバーレイ描画（セーフライン外側オーバーレイ／コマ枠線／白フチ／作品情報／ノンブル／画像レイヤー／コマサムネイル）は**`bpy.ops.render.render()` の出力画像には焼き込まれない**ため、書き出し時は以下のパイプラインで画像を合成する:

1. **コマの3Dレンダリング** — 各コマの `cNN.blend` をEEVEE-Nextまたは Workbench で高解像度レンダリング。書き出し用キャッシュに保存（`_preview.png` を流用可）
2. **紙面ベース画像の生成** — 用紙サイズで空の画像（白背景）をPillowで作成
3. **画像レイヤー合成** — Pillowで画像レイヤーを重ね描画（3.1.1と同じロジックをPillow版で再実装）
4. **コマ内レンダー画像の貼り込み** — 各コマのレンダー結果をコマ枠の位置・形状に合わせて貼る（Z順序・重なりくり抜きも反映）
5. **Grease Pencil v3 レイヤーの合成** — 各レイヤーを書き出し解像度でレンダリングしてPillowで重ねる
6. **コマ枠線・白フチの描画** — Pillowでベクター描画
7. **フキダシ・テキストの描画** — Pillow + fontToolsでテキスト合成（ビューポートの `blf` と同じロジックを再実装）
8. **作品情報・ノンブルの描画** — 同上
9. **トンボの描画**（オプション）
10. **セーフライン外側オーバーレイは書き出しに含めない**
11. **カラーモード変換** — モノクロ/グレースケール/CMYK への変換（CMYK時は littleCMS でICCプロファイル変換）
12. **ファイル保存** — PNG/JPEG/TIFF/PSD/PDF

書き出し時の描画関数とビューポート用 `draw_handler` 関数は**同じ計算ロジックを共有**するよう設計し、`typography/metrics.py` / `ui/overlay_shared.py` のような共通層を用意する。

#### 3.8.4a PSD レイヤー構成
- 用紙色、画像レイヤー、コマ内容、Grease Pencil、効果線、フキダシ、テキスト、トンボ、作品情報、ノンブルを**別レイヤー**で保持する
- 各コマは `panel/<panel>/content` の階層で保持し、**コマ形状のレイヤーマスクは content グループに掛ける**
- 白フチと枠線はコママスクで切れないよう、content の外に分けて保持する
- Grease Pencil / 効果線 / フキダシ / テキストは **ラスターレイヤー** として出力する
- `.clip` 直接出力は当面見送り、**PSD を主系統**にする

## 4. 保存形式

### 4.1 階層構造（確定）

B-Nameの管理単位は以下の3層:

```
作品 ＞ ページ ＞ コマ
```

「プロジェクト」という呼称は使わない。各単位の実体:

- **作品**: `MyWork.bname/` ディレクトリ（拡張子風のフォルダ名）
- **ページ**: `pNNNN/` 配下の1ディレクトリ
- **コマ**: `cNN.blend`（独立ファイル）

### 4.2 採用方針: 作品フォルダ + 独立コマファイル方式（確定）
複数の.blendファイルを扱う必要があり、かつ**1コマ=1.blendファイル**として独立させる。

```

├── work.blend                      # 紙面編集モードのメイン .blend

│                                   #   ・コマ間隔ルール（上下7.3mm / 左右2.1mm 既定）
```

- グローバルアセットライブラリ例: `D:\Develop\Blender\B-Name-Assets\`

### 4.3 用紙プリセットの保存場所
用紙プリセットは2層で管理:

1. **グローバルプリセット**: アドオン設定配下（全作品から参照可能）
   例: `<Blenderユーザー設定>/scripts/addons/B-Name/presets/paper/`
   既定プリセット「集英社マンガ誌汎用」はここに同梱
2. **作品ローカルプリセット**: `MyWork.bname/assets/templates/`
   作品固有の用紙設定を保存

開始ページの既定値は **左から** とする。

### 4.4 ファイル命名規則
- **ページディレクトリ**: `pNNNN/`（`p` + 4桁ゼロパディング、例: `p0001/`）
- **コマファイル**: `cNN.blend` / `cNN.json` / `cNN_thumb.png`（スクショ）／`cNN_preview.png`（任意の高品質プレビュー）（`c` + 2桁ゼロパディング、例: `c01.blend`、最大 `c99`）
  - `_thumb.png` は初回コマ編集モード終了時に生成、それまでは不在
  - `_preview.png` はユーザー手動生成時のみ存在
- 内部のファイル名は常に連番（リネームしない）。ユーザー向けの表示名は `cNN.json` に別途保持
  - これによりユーザーが表示名を変更しても参照関係が壊れない
- 新規コマ追加時は空いている最小の番号を採番
- コマ移動・複製時、移動先で番号が衝突したら移動先の空き番号に自動リネーム

### 5.2 依存関係ルール
- `panels/` `operators/` は `core/` `utils/` に依存してよい
- `core/` は `utils/` にのみ依存
- `utils/` は Blender API とPython標準ライブラリのみ依存
- 逆方向の依存は禁止（循環回避）

### 5.3 `register()` / `unregister()` 順序
各サブモジュールには `register()` / `unregister()` を定義し、`__init__.py` は以下の順で呼び出す。循環importと `PropertyGroup` の前方参照問題を回避する:

1. `preferences`
2. `core`（データ構造・PropertyGroup）
3. `io`（ファイル入出力）
4. `typography`
5. `operators`
6. `panels`
7. `ui`
8. `keymap`（最後、他モジュール登録後にキー割り当て）

`unregister()` は逆順で呼び出す。

`PropertyGroup` の相互参照は `PointerProperty` の循環定義を避け、`StringProperty`（ID参照）+ lookup関数で解決する。

## 8. 技術方針（確定済み）

プログラミング詳細の判断はユーザーに委ねず、以下の方針で確定する。開発中に重大な問題が発覚した場合のみ、代替案への変更をユーザーに相談する。

### 8.1 Meldex受信方法: ローカルHTTPサーバー方式
**採用**: B-NameアドオンがBlender内で軽量HTTPサーバー（localhost限定）を起動し、Meldexの「B-Nameへ送信」ボタンがそのエンドポイントにJSONをPOSTする。

理由:
- ユーザー操作が1ボタンで完結（ファイル保存→読み込みの手作業不要）
- 構造化データ（ルビ・縦中横含む）を確実に受け渡せる
- 将来的な双方向拡張にも対応しやすい

実装方式（スレッドセーフ対応）:
- `http.server.ThreadingHTTPServer` を**別スレッドで起動**
- 受信ハンドラは `bpy.data` を直接触らず、**受信したJSONを `queue.Queue` に積むだけ**
- **`bpy.app.timers.register(poll_queue, first_interval=0)`** でメインスレッドから定期的にキューを処理し、そこで `bpy.data` を更新
- これによりBlenderの「非メインスレッドからの bpy 呼び出し禁止」制約を回避
- アドオン無効化時（`unregister`）: サーバー `shutdown()` → スレッド `join()` → timer解除 を確実に実行
- ポート衝突時は連番で代替ポートをフォールバック試行

安全策:
- localhost以外からの接続を拒否
- ポートはアドオン設定で変更可能（既定値: 47817 等、衝突しにくい番号）
- Blender起動中のみ受信可能（Blender未起動時はMeldex側でエラー表示）

### 8.2 ブラシ質感: Grease Pencil標準ベース → 不足分のみ拡張
**採用**: まずGrease Pencil標準ブラシで漫画的線質（入り抜き・筆圧・テーパー）を構築。不足する質感（かすれ・アナログ感）は Phase 3 で独自ブラシプリセットを追加。

独自ブラシエンジンの実装は Phase 6 以降に後送り。

### 8.3 画像レイヤー: `draw_handler_add` + `gpu` オーバーレイ方式で実装
**採用（必要と確定）**: 3.1.1 に記載の通り実装。平面メッシュ方式は採用しない。

### 8.4 パフォーマンス対策
- 紙面編集の正本は `work.blend` と JSON 群で保持し、コマ編集時のみ対象 `cNN.blend` を開く
- **コマは独立ファイル方式**のため、編集中のコマのみメモリに載せ、それ以外は `cNN_thumb.png` / `cNN_preview.png` のうち更新日時が新しい方をコマ枠にテクスチャ表示。未生成時はコマ枠の背景色（白）のみ
- レンダーキャッシュ画像はコマ編集モード終了時に更新
- サムネイルは各ページ/コマのJSONにキャッシュパスを保持

### 8.5 組版エンジンのフォント処理: fontTools同梱方式
**採用**: Pythonの `fontTools` ライブラリをアドオンに同梱し、OpenType `vert` フィーチャで縦書き用字形に切り替える。

理由:
- Blender標準APIでは字形切り替えが限定的
- `fontTools` はMITライセンスで同梱可能
- ルビ・縦中横でも同じフォントメトリクスを使えるため実装が統一できる

代替案（`fontTools` が重すぎる場合）: 主要フォントの字形マップを事前生成してJSON同梱。

### 8.6 3Dアセット管理: Blenderアセットブラウザ活用（独自UI不要）
**採用**: B-Name独自のアセット管理UIは作らず、Blender標準のアセットブラウザをそのまま利用する。

2層のアセットライブラリ（3.4.4 参照）をユーザーがBlender設定で登録し、コマ編集モード中のシーンへドラッグ&ドロップでリンク参照を追加する。

B-Nameが提供するのは以下の連携機能のみ:
- アドオン設定画面に「アセットライブラリ登録ガイド」を表示
- ドラッグ&ドロップを受け取って、コマの `cNN.json` にリンク参照情報を記録
- カスタム右クリックメニュー「リンク元ファイルを開く」

### 8.7 リンク元ファイルを開く: subprocessで新Blenderインスタンス起動
**採用**: `右クリック > B-Name > リンク元ファイルを開く` で新しいBlenderインスタンスを `subprocess` 起動する（既定動作）。

理由:
- 現在の編集作業を失わない
- B-Name作品を開いたまま、別Blenderでモデル編集できる
- モデル側で保存後、B-Name側は次回コマレンダー時に自動反映

設定で「現在のBlenderで開く」にも切替可能。

### 8.9 配布・互換性（Extensions Platform対応）

### 8.10 画面回転時のカメラビュー固定
3.6のカメラリグ方式（カメラの視線軸ロール回転）は、ビューポートが**カメラビュー**になっている時のみ画面上で回転が見える。このため:

- B-Nameの**紙面編集モード中およびコマ編集モード中は、ビューポートを常にカメラビュー（`SpaceView3D.region_3d.view_perspective = 'CAMERA'`）に固定**
- ユーザーがテンキーなどでカメラビューから外れる操作をした場合、自動でカメラビューに戻す
- 「一時的に自由視点にする」モードは将来の拡張として保留

### 8.11 キーマップ既定オーバーライド
B-Name専用キーマップは `bpy.types.KeyMap` に登録するだけでは既定キーマップより優先されない場合がある。以下の二段構えで確実にオーバーライド:

- **B-Name作品を開いている時のみ**、関連する既定キーマップ項目（`Space` / `Ctrl+Space` / `Shift+Space` / 右クリック等）の `KeyMapItem.active = False` に切り替え
- アドオン無効化時・作品を閉じた時に元の `active` 状態へ復元
- Blenderのキーマップ Preset（Blender / Industry Compatible 等）で既定が異なるため、**現在のPresetを検出して対応マッピングを選択**するフォールバックUIをアドオン設定に用意
- ユーザー設定の `spacebar_action`（Tool / Search / Playback）は退避・復元する
- 右クリック（スポイト）は Preset 差異が大きいため、アドオン設定で「B-Nameモード中のみ右クリックをスポイトに割り当てる」スイッチを提供

### 8.12 サブプロセスでの新Blender起動（詳細）
3.4.5 / 8.7 の「リンク元ファイルを開く」の実装詳細:

```python
import subprocess, sys
from pathlib import Path
import bpy

def open_in_new_blender(blend_path: Path):
    kwargs = {"close_fds": True}
    if sys.platform == "win32":
        # Windowsでコンソール窓を出さない
        kwargs["creationflags"] = subprocess.CREATE_NO_WINDOW
    subprocess.Popen([bpy.app.binary_path, str(blend_path)], **kwargs)
```

- Windows: `CREATE_NO_WINDOW` でコンソール窓抑制
- macOS: `bpy.app.binary_path` は `.app` バンドル内の実行ファイル (`Blender.app/Contents/MacOS/Blender`) を直接指す。これをそのまま起動してよい（`open -a` は使わない）
- Linux: 環境変数継承は既定でOK
- 起動後のインスタンスでB-Nameを自動ロードするか否かはアドオン設定で選択

### 8.13 アセットブラウザ連携の記録方式
3.4.4 のドラッグ&ドロップでの「`cNN.json` へのリンク記録」実装:

- Blender 4.1以降で `bpy.types.FileHandler` が整備されており、5.xで機能拡張継続
- **D&D時のカスタム挙動を実装する場合**: `FileHandler.poll_drop` を使う
- **D&Dをフックできない場合のフォールバック**: `bpy.app.handlers.depsgraph_update_post` で新規Object出現を検知 → `object.library.filepath` を `cNN.json` の `linkedAssets` に記録
- 半自動運用: リンク後に右クリック → `B-Name > このリンクを記録` メニューで明示的に記録

### 8.8 コマのスクショサムネイル更新タイミング
- コマ編集モードから紙面編集モードへ戻る時に、**そのコマのビューポートをスクリーンショット**（`bpy.ops.screen.screenshot_area()` または `GPUOffScreen`）で取得し `cNN_thumb.png` に保存
- **レンダリングは行わない**（Freestyle・ライティング計算なし、数十ms程度で完了）
- スクショ解像度はコマ枠の表示サイズに合わせた最小限
- **新規コマで一度もスクショを取っていない場合**はコマ枠の背景色（白）のみ表示、プレースホルダ画像は生成しない
- ユーザーが「高品質プレビュー生成」をコマ右クリックから要求したときのみ低解像度レンダリングを実行し `cNN_preview.png` に保存
- 「全コマの高品質プレビューを再生成」も手動ボタンで可能
- 書き出し時（Phase 6）は高解像度で全コマを再レンダリング（書き出し結果に反映）

**技術方針**（Blender 4.3+ / 5.x 基準）:
- 対応Blenderバージョン: 下限 4.3.0、主ターゲット 5.x（8.9.3）
- 配布形式: Extensions Platform（`blender_manifest.toml` + `wheels/` 同梱、`bl_info` 最小化）（8.9）
- Grease Pencil v3 前提（v2 API不使用、レイヤーフォルダは `GreasePencilLayerGroup` をそのまま使用）（3.1.1 / 10章）
- コマ枠画像・画像レイヤー・オーバーレイ要素の描画: `SpaceView3D.draw_handler_add` + `gpu` モジュール（シェーディングモード非依存）（3.4.3a）
- 書き出しパイプライン: Pillowでオーバーレイ要素を再描画して合成（書き出し画像に焼き込む）（3.8.4）
- 組版エンジン: 3層構成 `blf` + `fontTools` + `Pillow`（3.1.5）
- Meldex受信: ローカルHTTPサーバー方式（threading + queue + `bpy.app.timers` でメインスレッド復帰）（8.1）
- ブラシ: Grease Pencil v3 標準ベース、不足分は後段で拡張（8.2）
- 3Dアセット管理: Blender標準アセットブラウザを活用、独自UIは作らない。D&D記録は `FileHandler` または `depsgraph_update_post`（3.4.4 / 8.6 / 8.13）
- アセットライブラリ: 2層構造（グローバル / 作品固有）（3.4.4）
- リンク元ファイルを開く: 新Blenderインスタンスをsubprocess起動（Windows: CREATE_NO_WINDOW）（8.7 / 8.12）
- スクショサムネイル: コマ編集モード終了時に `cNN_thumb.png` をビューポートスクショで更新（第1候補: `bpy.ops.screen.screenshot_area()` + temp_override、フォールバック: `GPUOffScreen.draw_view3d()`）（3.4.3 / 8.8）
- 手動高品質プレビュー: ユーザー要求時のみ低解像度レンダリングし `cNN_preview.png` 保存（3.4.3）
- 新規コマで未生成時: コマ枠は背景色（白）のみ表示（3.4.3）
- 画面回転: カメラリグのロール回転（ビューポートはB-Nameモード中は常にカメラビュー固定）（8.10）
- キーマップ既定オーバーライド: B-Nameモード中のみ既定 `KeyMapItem.active = False`、Preset差異対応（8.11）
- register順序: preferences → core → io → typography → operators → panels → ui → keymap（unregister逆順）（5.3）

**ファイル命名規則**:
- ページディレクトリ: `pNNNN/`（`p` + 4桁ゼロパディング、例: `p0001/`）
- コマファイル: `cNN.blend` / `cNN.json` / `cNN_thumb.png` / `cNN_preview.png`（`c` + 2桁ゼロパディング、例: `c01.blend`、最大 `c99`）
- 内部ファイル名は常に連番固定、ユーザー向け表示名は `cNN.json` に別途保持

**開発体制**:
- B-NameはMeldexから独立したBlenderアドオン
- 連携はMeldex→B-Name一方向のみ
- Meldex側には「B-Nameへ送信」ボタンの追加が必要（別計画書で管理）
